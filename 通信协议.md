# 任务执行进度通信协议 v2

## 概述

本协议定义了 **发送端（研发管理系统）** 与 **接收端（SmartDosing 设备）** 之间的完整通信机制，包括设备配对、任务下发、进度跟踪。

### 设计原则

1. **零配置**：发送端发起配对 → 设备屏幕点击确认 → 自动交换密钥，无需手动填写任何密钥
2. **双通道通信**：接收端主动回调 + 发送端轮询，确保可靠性
3. **兼容现有架构**：在现有 Ktor 服务器和 API Key 体系上扩展

### 角色定义

| 角色 | 说明 | 服务类型 |
|------|------|---------|
| **发送端** | 研发管理系统，下发任务 | HTTP 客户端 → 调用接收端 API |
| **接收端** | SmartDosing 设备，执行配料 | Ktor HTTP 服务器（端口 8080） |

## 一、设备配对（首次连接）

配对流程类似蓝牙配对：发送端发起请求 → 设备屏幕弹窗 → 操作员点击确认 → 双方交换密钥。**配对只需一次，后续自动认证。**

### 1.1 发送端发起配对

#### `POST /api/device/pair`

> 此接口**无需认证**，任何局域网设备都可以发起配对请求。

**请求体**：

```json
{
  "senderUID": "SENDER-A1B2C3D4",
  "senderName": "研发管理系统",
  "senderIP": "192.168.1.100",
  "senderPort": 3000,
  "callbackBaseUrl": "http://192.168.1.100:3000",
  "timestamp": 1707350400000
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `senderUID` | string | 是 | 发送端唯一标识 |
| `senderName` | string | 是 | 发送端名称（显示在确认弹窗上） |
| `senderIP` | string | 是 | 发送端 IP |
| `senderPort` | integer | 是 | 发送端服务端口 |
| `callbackBaseUrl` | string | 是 | 发送端回调基础地址 |
| `timestamp` | long | 是 | 请求时间戳 |

**响应体**（立即返回，不等待用户确认）：

```json
{
  "success": true,
  "message": "配对请求已发送，请在设备屏幕上确认",
  "pairingId": "PAIR-20240208-XXXX",
  "expiresIn": 60
}
```

| 字段 | 说明 |
|------|------|
| `pairingId` | 本次配对请求 ID，用于轮询配对结果 |
| `expiresIn` | 配对请求有效期（秒），超时未确认自动失效 |

### 1.2 设备屏幕确认

接收端收到配对请求后：

1. 屏幕弹出确认对话框，显示发送端名称和 IP
2. 操作员点击 **「确认配对」** 或 **「拒绝」**
3. 确认后，接收端自动生成一个专属 API Key 给该发送端

### 1.3 发送端轮询配对结果

#### `GET /api/device/pair/{pairingId}`

> 此接口**无需认证**。

**响应 — 等待中**：

```json
{
  "success": true,
  "status": "WAITING",
  "message": "等待设备确认"
}
```

**响应 — 配对成功**：

```json
{
  "success": true,
  "status": "APPROVED",
  "message": "配对成功",
  "data": {
    "apiKey": "a3f8b2c1-xxxx-yyyy-zzzz",
    "deviceUID": "SD-12345678",
    "deviceName": "1号配料机",
    "deviceIP": "192.168.1.50",
    "devicePort": 8080
  }
}
```

> **关键**：`apiKey` 是接收端为该发送端生成的专属密钥。发送端保存后，后续所有 API 调用都携带此 Key，无需再次配对。

**响应 — 配对拒绝**：

```json
{
  "success": true,
  "status": "REJECTED",
  "message": "设备拒绝了配对请求"
}
```

**响应 — 已过期**：

```json
{
  "success": true,
  "status": "EXPIRED",
  "message": "配对请求已过期"
}
```

### 1.4 配对后的认证方式

配对成功后，发送端所有后续请求携带获得的 API Key：

```
X-API-Key: a3f8b2c1-xxxx-yyyy-zzzz
```

与现有认证体系完全一致，无需引入 JWT 或其他机制。

### 1.5 配对流程图

```
发送端                                    接收端 (SmartDosing)
  │                                          │
  │  POST /api/device/pair                   │
  │  {senderName, senderIP, ...}             │
  │ ──────────────────────────────────────►   │
  │                                          │  ┌──────────────────┐
  │   ◄──────────────────────────────────    │  │  屏幕弹出确认框  │
  │  {pairingId, expiresIn: 60}              │  │                  │
  │                                          │  │  "研发管理系统"   │
  │  GET /api/device/pair/{pairingId}        │  │  192.168.1.100   │
  │ ──────────────────────────────────────►   │  │                  │
  │   ◄──────────────────────────────────    │  │ [确认]    [拒绝] │
  │  {status: "WAITING"}                     │  └──────────────────┘
  │                                          │
  │  ... 轮询 ...                            │  操作员点击「确认」
  │                                          │  生成专属 apiKey
  │  GET /api/device/pair/{pairingId}        │
  │ ──────────────────────────────────────►   │
  │   ◄──────────────────────────────────    │
  │  {status: "APPROVED", apiKey: "xxx"}     │
  │                                          │
  │  ✅ 保存 apiKey，配对完成                  │
  │  后续请求携带 X-API-Key: xxx              │
```

## 二、任务下发（已配对后）

### `POST /api/transfer/task`

**认证方式**：`X-API-Key`（配对时获得的密钥）

在现有请求体基础上，**无需额外字段**。接收端已经在配对时保存了发送端的 `callbackBaseUrl`，回调地址自动拼接：

```
回调地址 = {callbackBaseUrl}/api/lan/devices/task-progress
```

请求体与现有格式完全一致：

```json
{
  "schemaVersion": "1.0",
  "transferId": "TF-1707350400000-A1B2C3D4",
  "sender": {
    "uid": "SENDER-A1B2C3D4",
    "name": "研发管理系统",
    "ip": "192.168.1.100",
    "appVersion": "2.0.0"
  },
  "task": { ... },
  "recipe": { ... }
}
```

> 发送端不需要填 `callbackUrl` 或 `callbackToken`，这些信息在配对阶段已经交换完毕。

## 四、接收端主动回调（推送模式）

任务状态变更时，接收端自动回调发送端。回调地址和认证信息来自配对阶段，无需每次任务都传。

### 回调地址

```
{配对时保存的 callbackBaseUrl}/api/lan/devices/task-progress
```

### 回调认证

```
X-API-Key: {接收端自己的 apiKey}
```

> 双方互持对方的 API Key：发送端持有接收端的 Key 调接收端接口，接收端持有发送端在配对时交换的 Key 做回调认证。
>
> **简化方案**：如果发送端不需要验证回调来源，接收端回调时可以不带认证头，发送端只校验请求体中的 `deviceUID` 即可。具体由发送端决定。

### 回调请求体

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `transferId` | string | **是** | 传输流水号 |
| `deviceUID` | string | **是** | 接收端设备 UID |
| `deviceName` | string | **是** | 接收端设备名称 |
| `status` | string | **是** | 执行状态（见枚举值） |
| `progress` | integer | 否 | 执行进度百分比 0-100 |
| `currentStep` | integer | 否 | 当前步骤序号（从 1 开始） |
| `totalSteps` | integer | 否 | 总步骤数（等于材料数量） |
| `currentMaterial` | string | 否 | 当前正在处理的材料名称 |
| `message` | string | 否 | 进度说明或错误原因 |
| `timestamp` | long | **是** | 上报时间戳（毫秒） |

### status 枚举值

| 值 | 含义 | 触发时机 | 说明 |
|----|------|---------|------|
| `ACCEPTED` | 已接收 | 操作员确认接收任务 | 任务进入待执行队列 |
| `REJECTED` | 已拒绝 | 操作员拒绝任务 | **终态** |
| `EXECUTING` | 执行中 | 开始配料操作 | 可携带 progress 等进度信息 |
| `COMPLETED` | 已完成 | 全部材料配料完成 | **终态**，progress 自动设为 100 |
| `EXEC_FAILED` | 执行失败 | 配料过程出错 | **终态**，message 应填写错误原因 |

### 回调示例

#### 1. 任务已接收

```json
{
  "transferId": "TF-1707350400000-A1B2C3D4",
  "deviceUID": "SD-12345678",
  "deviceName": "1号配料机",
  "status": "ACCEPTED",
  "message": "任务已接收，等待执行",
  "timestamp": 1707350450000
}
```

#### 2. 开始执行

```json
{
  "transferId": "TF-1707350400000-A1B2C3D4",
  "deviceUID": "SD-12345678",
  "deviceName": "1号配料机",
  "status": "EXECUTING",
  "progress": 0,
  "currentStep": 1,
  "totalSteps": 7,
  "currentMaterial": "基液",
  "message": "开始执行配方任务",
  "timestamp": 1707350500000
}
```

#### 3. 执行完成

```json
{
  "transferId": "TF-1707350400000-A1B2C3D4",
  "deviceUID": "SD-12345678",
  "deviceName": "1号配料机",
  "status": "COMPLETED",
  "progress": 100,
  "currentStep": 7,
  "totalSteps": 7,
  "message": "配方执行完成",
  "timestamp": 1707350800000
}
```

#### 4. 执行失败

```json
{
  "transferId": "TF-1707350400000-A1B2C3D4",
  "deviceUID": "SD-12345678",
  "deviceName": "1号配料机",
  "status": "EXEC_FAILED",
  "progress": 42,
  "currentStep": 3,
  "totalSteps": 7,
  "currentMaterial": "香精A",
  "message": "材料不足，称量中断",
  "timestamp": 1707350600000
}
```

### 回调失败处理

| 策略 | 说明 |
|------|------|
| 重试次数 | 最多重试 3 次 |
| 重试间隔 | 5s → 15s → 30s（指数退避） |
| 超时时间 | 单次请求 10 秒超时 |
| 兜底方案 | 回调失败不影响本地任务执行，发送端可通过轮询接口兜底 |

## 五、心跳接口扩展

现有 `POST /api/device/ping` 响应中增加当前任务执行状态：

```json
{
  "success": true,
  "message": "pong",
  "data": {
    "uid": "SD-12345678",
    "deviceName": "1号配料机",
    "status": "BUSY",
    "currentTask": {
      "transferId": "TF-1707350400000-A1B2C3D4",
      "status": "EXECUTING",
      "progress": 45,
      "currentStep": 3,
      "totalSteps": 7
    },
    "timestamp": 1707350500000
  }
}
```

> `status`：执行任务期间为 `BUSY`，空闲时为 `IDLE`。`currentTask` 仅在 `BUSY` 时存在。

## 七、接收端实现要点

### 7.1 数据库扩展

**`authorized_senders` 表新增字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `sender_port` | INTEGER | 发送端服务端口 |
| `callback_base_url` | TEXT | 发送端回调基础地址 |
| `sender_api_key` | TEXT | 为该发送端生成的专属 API Key |

**`received_tasks` 表新增字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `exec_status` | TEXT | 执行状态：PENDING/ACCEPTED/REJECTED/EXECUTING/COMPLETED/EXEC_FAILED |
| `progress` | INTEGER | 进度百分比 0-100，默认 0 |
| `current_step` | INTEGER | 当前步骤序号，默认 0 |
| `total_steps` | INTEGER | 总步骤数，默认 0 |
| `current_material` | TEXT | 当前材料名称（可空） |
| `exec_message` | TEXT | 执行消息（可空） |
| `exec_updated_at` | INTEGER | 执行状态最后更新时间 |

### 7.2 配对请求管理

新增内存缓存 `PairingRequestStore`：

```
Map<pairingId, PairingSession>

PairingSession:
  - senderUID, senderName, senderIP, senderPort
  - callbackBaseUrl
  - status: WAITING / APPROVED / REJECTED / EXPIRED
  - generatedApiKey: 确认后生成
  - createdAt: 创建时间
  - expiresAt: 过期时间（createdAt + 60s）
```

- 配对请求存内存即可，不需要持久化
- 60 秒未确认自动过期清理
- 同一 senderUID 同时只允许一个配对请求

### 7.3 状态流转规则

```
PENDING ──► ACCEPTED ──► EXECUTING ──► COMPLETED
   │                         │
   ▼                         ▼
REJECTED                 EXEC_FAILED
```

- `REJECTED`、`COMPLETED`、`EXEC_FAILED` 为终态，不可再变更
- `EXECUTING` 状态下可多次更新 progress 等进度字段（不触发回调，仅更新本地供轮询）
- 仅状态变更时触发回调

### 7.4 回调触发时机

| 状态变更 | 是否触发回调 | 说明 |
|---------|------------|------|
| PENDING → ACCEPTED | 是 | 操作员确认接收 |
| PENDING → REJECTED | 是 | 操作员拒绝 |
| ACCEPTED → EXECUTING | 是 | 开始配料 |
| EXECUTING → COMPLETED | 是 | 配料完成 |
| EXECUTING → EXEC_FAILED | 是 | 配料失败 |
| EXECUTING 进度更新 | **否** | 仅更新本地，发送端通过轮询获取 |

### 7.5 认证拦截器调整

现有 API Key 认证逻辑需要调整：

- 现有全局 `apiKey` 继续用于 Web UI 页面
- 新增：检查请求携带的 Key 是否匹配任意一个已配对发送端的 `sender_api_key`
- `/api/device/pair` 和 `/api/device/pair/{id}` 路径**跳过认证**

```
认证逻辑：
  if (path == "/api/device/pair*") → 放行
  if (requestKey == globalApiKey) → 放行（Web UI）
  if (requestKey in authorizedSenderApiKeys) → 放行（已配对发送端）
  else → 401 Unauthorized
```

## 八、与原协议的对照

| 原协议设计 | 本协议设计 | 变更原因 |
|-----------|-----------|---------|
| Device JWT Token 认证 | 配对授权 + API Key | 操作员屏幕确认，零手动配置 |
| 需要手动填写密钥 | 配对自动交换密钥 | 用户体验更好 |
| 设备 POST 到固定服务端地址 | 配对时交换回调地址 | 不依赖固定地址 |
| 仅 3 次上报 | 4 次回调 + 轮询兜底 | 覆盖完整任务生命周期 |
| 无轮询接口 | 新增进度查询接口 | 回调失败时的兜底方案 |
| 无心跳关联 | 心跳携带当前任务信息 | 发送端可感知设备状态 |

## 九、安全考虑

1. **配对接口限制**：可限制同一 IP 每分钟最多发起 5 次配对请求，防止滥用
2. **API Key 隔离**：每个发送端有独立的 API Key，撤销某个发送端不影响其他
3. **局域网限制**：所有接口仅在局域网内可访问，不暴露到公网
4. **配对过期**：未确认的配对请求 60 秒自动失效
5. **发送端管理**：设备端可在设置页面查看、禁用、删除已配对的发送端
