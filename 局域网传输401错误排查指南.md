# 局域网传输 HTTP 401 错误排查指南

> **面向对象**: 后台开发团队
> **设备端版本**: SmartDosing Android v2.0 (commit: c18be0c)
> **更新日期**: 2026-02-27

---

## 一、认证机制概述

SmartDosing 设备端采用 **配对授权 + API Key** 认证机制，所有 `/api` 路径下的接口（除配对接口外）都需要携带有效的 `X-API-Key` 请求头。

### 认证流程

```
┌─────────────┐                          ┌──────────────┐
│  后台系统   │                          │ SmartDosing  │
│ (发送端)    │                          │   (接收端)   │
└─────────────┘                          └──────────────┘
       │                                        │
       │  1. POST /api/device/pair              │
       │  (无需认证)                             │
       ├───────────────────────────────────────►│
       │                                        │ 屏幕弹窗确认
       │  2. 返回 pairingId                      │
       │◄───────────────────────────────────────┤
       │                                        │
       │  3. GET /api/device/pair/{pairingId}   │
       │  (轮询配对结果，无需认证)                │
       ├───────────────────────────────────────►│
       │                                        │
       │  4. 返回 apiKey (配对成功后)            │
       │◄───────────────────────────────────────┤
       │                                        │
       │  5. POST /api/transfer/task            │
       │  Header: X-API-Key: {apiKey}           │
       ├───────────────────────────────────────►│
       │                                        │ ✅ 认证通过
       │  6. 返回任务接收结果                     │
       │◄───────────────────────────────────────┤
```

## 二、401 错误的常见原因

### 原因 1: 未携带 X-API-Key 请求头 ⭐⭐⭐⭐⭐

**现象**: 直接返回 401，响应体为:
```json
{
  "success": false,
  "message": "未授权：请提供有效的 API Key"
}
```

**排查方法**:
```bash
# 错误示例 (缺少请求头)
curl -X POST http://192.168.1.50:8080/api/transfer/task \
  -H "Content-Type: application/json" \
  -d '{"schemaVersion":"1.0",...}'

# 正确示例 (携带 X-API-Key)
curl -X POST http://192.168.1.50:8080/api/transfer/task \
  -H "Content-Type: application/json" \
  -H "X-API-Key: a3f8b2c1-xxxx-yyyy-zzzz" \
  -d '{"schemaVersion":"1.0",...}'
```

**解决方案**:
- 确保所有 `/api` 路径的请求都携带 `X-API-Key` 请求头
- 请求头名称必须严格匹配 `X-API-Key` (区分大小写)

---

### 原因 2: API Key 值错误或无效 ⭐⭐⭐⭐

**现象**: 携带了请求头但仍返回 401

**排查方法**:
1. **检查 Key 格式**: API Key 应为 UUID 格式 (如 `a3f8b2c1-1234-5678-90ab-cdef12345678`)
2. **检查 Key 来源**: 必须通过配对流程获取，不能自己生成
3. **检查 Key 有效性**:
   ```bash
   # 测试 Key 是否有效 (调用心跳接口)
   curl -H "X-API-Key: your-key-here" \
     http://192.168.1.50:8080/api/device/ping

   # 如果返回 401，说明 Key 无效
   # 如果返回 200，说明 Key 有效，问题在其他地方
   ```

**解决方案**:
- 重新执行配对流程获取新的 API Key
- 检查后台系统是否正确保存了配对返回的 `apiKey` 字段

---

### 原因 3: 未完成配对流程 ⭐⭐⭐⭐⭐

**现象**:
- 直接调用 `/api/transfer/task` 返回 401
- 或者返回 400，错误码为 `UNAUTHORIZED_SENDER`

**排查方法**:
```bash
# 步骤 1: 发起配对请求
curl -X POST http://192.168.1.50:8080/api/device/pair \
  -H "Content-Type: application/json" \
  -d '{
    "senderUID": "BACKEND-001",
    "senderName": "研发管理系统",
    "senderIP": "192.168.1.100",
    "senderPort": 3000,
    "callbackBaseUrl": "http://192.168.1.100:3000",
    "timestamp": 1707350400000
  }'

# 响应示例:
# {
#   "success": true,
#   "message": "配对请求已发送，请在设备屏幕上确认",
#   "data": {
#     "pairingId": "PAIR-1707350400000-A1B2",
#     "expiresIn": 60
#   }
# }

# 步骤 2: 在设备屏幕上点击「确认配对」

# 步骤 3: 轮询配对结果 (每 2-3 秒轮询一次)
curl http://192.168.1.50:8080/api/device/pair/PAIR-1707350400000-A1B2

# 等待中:
# {"success":true,"data":{"status":"WAITING","message":"等待设备确认"}}

# 配对成功:
# {
#   "success": true,
#   "data": {
#     "status": "APPROVED",
#     "message": "配对成功",
#     "apiKey": "a3f8b2c1-xxxx-yyyy-zzzz",  ← 保存此 Key
#     "deviceUID": "SD-12345678",
#     "deviceName": "1号配料机",
#     "deviceIP": "192.168.1.50",
#     "devicePort": 8080
#   }
# }

# 步骤 4: 使用获取的 apiKey 调用任务接口
curl -X POST http://192.168.1.50:8080/api/transfer/task \
  -H "Content-Type: application/json" \
  -H "X-API-Key: a3f8b2c1-xxxx-yyyy-zzzz" \
  -d '{"schemaVersion":"1.0",...}'
```

**注意事项**:
- 配对请求有效期为 **60 秒**，超时需重新发起
- 配对必须在设备屏幕上手动确认，无法自动通过
- 同一 `senderUID` 只能有一个活跃配对请求

---

### 原因 4: 发送端被禁用或撤销授权 ⭐⭐

**现象**: 之前能正常调用，突然返回 401

**可能原因**:
- 设备端操作员在「设置 → 已授权设备」中禁用或删除了该发送端
- 设备端数据库被清空或重置

**排查方法**:
```bash
# 测试当前 Key 是否有效
curl -H "X-API-Key: your-key-here" \
  http://192.168.1.50:8080/api/device/ping

# 如果返回 401，说明授权已被撤销
```

**解决方案**:
- 联系设备端操作员确认授权状态
- 重新执行配对流程

---

### 原因 5: 请求头名称拼写错误 ⭐⭐⭐

**常见错误**:
```bash
# ❌ 错误示例
-H "X-Api-Key: xxx"        # 小写 Api
-H "Api-Key: xxx"          # 缺少 X-
-H "X-API-KEY: xxx"        # 全大写 KEY
-H "Authorization: xxx"    # 错误的请求头名称

# ✅ 正确示例
-H "X-API-Key: xxx"        # 严格匹配
```

**排查方法**:
- 使用抓包工具 (如 Wireshark, Charles) 查看实际发送的请求头
- 检查代码中请求头名称是否为 `X-API-Key`

## 三、快速排查步骤

### Step 1: 确认配对状态

```bash
# 调用心跳接口测试认证
curl -v -H "X-API-Key: your-key-here" \
  http://192.168.1.50:8080/api/device/ping

# 预期响应 (200 OK):
# {
#   "success": true,
#   "message": "pong",
#   "data": {
#     "uid": "SD-12345678",
#     "deviceName": "1号配料机",
#     "status": "IDLE",
#     "timestamp": 1707350400000
#   }
# }
```

**结果判断**:
- ✅ 返回 200: API Key 有效，问题在其他地方
- ❌ 返回 401: API Key 无效或未配对，执行 Step 2

---

### Step 2: 重新配对

```bash
# 1. 发起配对
curl -X POST http://192.168.1.50:8080/api/device/pair \
  -H "Content-Type: application/json" \
  -d '{
    "senderUID": "YOUR-BACKEND-UID",
    "senderName": "您的系统名称",
    "senderIP": "您的服务器IP",
    "senderPort": 3000,
    "callbackBaseUrl": "http://您的服务器IP:端口",
    "timestamp": '$(date +%s000)'
  }'

# 2. 记录返回的 pairingId

# 3. 通知设备端操作员在屏幕上点击「确认配对」

# 4. 轮询配对结果 (替换 {pairingId})
curl http://192.168.1.50:8080/api/device/pair/{pairingId}

# 5. 配对成功后保存返回的 apiKey
```

---

### Step 3: 验证任务接口

```bash
# 使用新获取的 API Key 调用任务接口
curl -X POST http://192.168.1.50:8080/api/transfer/task \
  -H "Content-Type: application/json" \
  -H "X-API-Key: {新获取的apiKey}" \
  -d '{
    "schemaVersion": "1.0",
    "transferId": "TF-'$(date +%s000)'-TEST",
    "senderUID": "YOUR-BACKEND-UID",
    "senderName": "您的系统名称",
    "timestamp": '$(date +%s000)',
    "task": {
      "title": "测试任务",
      "recipeCode": "TEST-001",
      "recipeName": "测试配方",
      "quantity": 100,
      "unit": "g",
      "priority": "NORMAL"
    },
    "recipe": {
      "code": "TEST-001",
      "name": "测试配方",
      "totalWeight": 100,
      "materials": [
        {
          "name": "材料A",
          "weight": 50,
          "unit": "g",
          "sequence": 1
        },
        {
          "name": "材料B",
          "weight": 50,
          "unit": "g",
          "sequence": 2
        }
      ]
    }
  }'
```

**预期响应 (201 Created)**:
```json
{
  "success": true,
  "message": "任务接收成功",
  "data": {
    "success": true,
    "message": "任务接收成功",
    "transferId": "TF-1707350400000-TEST",
    "receivedTaskId": "RT-A1B2C3D4",
    "receiverUID": "SD-12345678",
    "receiverName": "1号配料机",
    "timestamp": 1707350400000,

  }

```

## 四、常见错误响应对照表

| HTTP 状态码 | 错误码 | 错误信息 | 原因 | 解决方案 |
|------------|--------|---------|------|---------|
| 401 | - | 未授权：请提供有效的 API Key | 未携带或 Key 无效 | 检查请求头，重新配对 |
| 400 | UNAUTHORIZED_SENDER | 发送端未授权，请先通过 /api/device/pair 完成配对 | 未完成配对流程 | 执行配对流程 |
| 400 | DUPLICATE_TRANSFER | 任务已接收过，请勿重复发送 | transferId 重复 | 使用新的 transferId |
| 400 | RECIPE_EXISTS | 配方编码已存在，如需覆盖请人工确认 | 配方已存在 | 更换配方编码或手动删除旧配方 |
| 400 | UNSUPPORTED_VERSION | 协议版本不受支持 | schemaVersion 错误 | 使用 "1.0" |
| 400 | FIELD_INVALID | 任务内容非法 | 字段校验失败 | 检查请求体格式 |
| 400 | FIELD_MISSING | 必填字段缺失 | 缺少必填字段 | 补充缺失字段 |
| 429 | - | 请求过于频繁，请稍后再试 | 配对请求限流 | 等待 1 分钟后重试 |

## 七、联系设备端支持

如果以上步骤无法解决问题，请提供以下信息联系设备端技术支持:

1. **错误现象**:
   - HTTP 状态码
   - 完整的错误响应体
   - 出现时间和频率

2. **请求信息**:
   - 完整的 curl 命令或 HTTP 请求报文
   - 是否携带 `X-API-Key` 请求头
   - API Key 的前 8 位字符 (如 `a3f8b2c1-...`)

3. **配对状态**:
   - 是否完成过配对流程
   - 配对时间
   - 设备端是否显示该发送端在「已授权设备」列表中

4. **设备信息**:
   - 设备 IP 地址
   - 设备 UID (如 `SD-12345678`)
   - 设备端应用版本

## 八、附录: 认证机制技术细节

### 认证拦截器源码位置
- 文件: `app/src/main/java/com/example/smartdosing/web/WebServerManager.kt`
- 行号: 158-198

### 数据库表结构
```sql
-- 授权发送端表
CREATE TABLE authorized_senders (
    uid TEXT PRIMARY KEY NOT NULL,
    name TEXT NOT NULL,
    ip_address TEXT,
    sender_port INTEGER,
    callback_base_url TEXT,
    sender_api_key TEXT,           -- 专属 API Key
    authorized_at INTEGER NOT NULL,
    last_task_at INTEGER,
    task_count INTEGER DEFAULT 0,
    app_version TEXT,
    is_active INTEGER DEFAULT 1    -- 0=禁用, 1=启用
);
```

### API Key 缓存机制
- 缓存有效期: 5 分钟
- 缓存失效后自动查询数据库
- 发送端被撤销后缓存会被清除
